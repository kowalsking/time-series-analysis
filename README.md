# Аналіз часових рядів повітряних тривог в Україні

Навчальний MVP: повний цикл аналізу часових рядів сповіщень про повітряну тривогу —
від сирих даних до інтерактивного Streamlit-дашборда з патернами та короткостроковим
прогнозом.

## Що показує дашборд

- **Огляд** — KPI (всього тривог, регіонів, годин тривог, медіанна тривалість),
  погодинний ряд активних тривог, денний тренд.
- **Патерни** — розподіли за годиною доби та днем тижня, теплова карта
  «регіон × година», топ регіонів, кореляція активності між регіонами.
- **Прогноз** — короткостроковий прогноз погодинної к-сті тривог
  (наївний сезонний baseline vs Holt-Winters) з метриками MAE/RMSE.

У бічній панелі — перемикач джерела даних (live API за замовчуванням / історичні),
фільтр регіонів і пресети періоду (усі дані / останній тиждень / місяць / власний діапазон).

## Стек

Python 3.11+ (перевірено на 3.14), pandas, numpy, statsmodels, plotly, streamlit,
requests, python-dotenv, pyarrow, pytest.

## Встановлення

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## Дані

Основне джерело — Kaggle-датасет
[`dimakyn/air-alarm-ukrain-2022022420220409`](https://www.kaggle.com/datasets/dimakyn/air-alarm-ukrain-2022022420220409)
(історія тривог по регіонах, ~24.02.2022–09.04.2022).

1. Завантажте CSV із Kaggle і покладіть у `data/raw/` (будь-яке ім'я `*.csv`).
2. Перевірте реальну схему колонок:
   ```bash
   python -m src.data_loader        # друкує колонки/типи/перші рядки
   ```
3. Якщо назви колонок відрізняються від очікуваних — додайте їх у
   `COLUMN_ALIASES` у `src/config.py`.

### Live-дані з API alerts.in.ua

Проєкт також працює з **реальними** даними через офіційний API alerts.in.ua.

1. Отримайте токен на сайті alerts.in.ua і покладіть у `.env`:
   ```
   ALERTS_TOKEN=ваш_токен
   ```
2. Синхронізуйте історію (тягне ~місяць по всіх областях):
   ```bash
   python -m src.ingest            # період з config.API_PERIOD (month_ago)
   python -m src.ingest week_ago   # або явно
   ```
3. У дашборді в бічній панелі оберіть джерело **«Live API (alerts.in.ua)»**
   (там же — кнопка «🔄 Синхронізувати з API»).

> ⚠️ **Ліміт API: 30 запитів / 10 хв.** Один sync = ~27 запитів (по областях),
> тому вкладається у вікно. Відповіді кешуються на диск (`data/processed/api_cache/`,
> TTL 30 хв), щоб не палити ліміт на повторах. На 429 клієнт чекає `Retry-After`
> (якщо це довше за `API_MAX_RETRY_WAIT_S` — повідомляє і виходить).

Для часових рядів беруться тривоги типу `air_raid`, зведені до рівня області
(`location_oblast`) — сумісно з історичним датасетом.

### Без Kaggle: синтетичні дані (dev-фолбек)

Щоб одразу запустити проєкт без завантаження:

```bash
python -m src.synth      # згенерує data/raw/alerts.csv
```

> ⚠️ Синтетичні дані відтворюють реалістичні патерни, але **не є реальними
> тривогами** — лише для розробки/демонстрації.

## Запуск

```bash
# 1) побудувати таблицю подій (кешується в data/processed/)
python -m src.preprocess

# 2) оцінити прогноз у консолі (опційно)
python -m src.forecast

# 3) запустити дашборд
streamlit run app/dashboard.py
```

## Тести

```bash
pytest
```

## Структура

```
src/
  config.py        # шляхи, tz, маппінг колонок, константи
  data_loader.py   # читання сирого CSV
  preprocess.py    # → канонічна таблиця подій [region, start_dt, end_dt, duration_min]
  features.py      # погодинні часові ряди
  analysis.py      # KPI, патерни, кореляції
  forecast.py      # seasonal_naive + Holt-Winters, метрики
  viz.py           # plotly-графіки
  synth.py         # генератор синтетичних даних (dev)
  api_client.py    # клієнт alerts.in.ua (rate-limit, кеш, 429)
  ingest.py        # sync історії з API → канонічна таблиця подій
app/dashboard.py   # Streamlit (перемикач: історичні / live API)
tests/             # pytest
```

## Обмеження

Період даних короткий в обох джерелах (Kaggle ~45 днів; API віддає лише ~місяць
історії й глибшої заднім числом не дає), тому **тижневу/довгу сезонність не аналізуємо** —
фокус на добовому циклі та короткостроковому горизонті прогнозу.

## Покращення (куди розвивати)

- **Накопичення історії з API.** API дає лише рухоме вікно ~місяць. Якщо кожен sync
  дописуватиме нові тривоги в постійне сховище (дедуп за `id`), датасет ростиме за межі
  місяця — і з'явиться матеріал для тижневої/сезонної аналітики.
- **Гранулярність до районів/громад.** Зараз API-дані зводяться до рівня області;
  `location_type` дозволяє аналізувати райони та окремі громади.
- **Сильніші моделі прогнозу.** SARIMAX, Prophet або ML (градієнтний бустинг із
  лаговими/календарними ознаками), екзогенні фактори, інтервали довіри, бектест на
  кількох вікнах. Зараз — baseline + Holt-Winters із добовою сезонністю.
- **Виявлення аномалій.** Окрема вкладка для детекції масованих атак (різкі синхронні
  сплески по багатьох регіонах) — частково вже видно через кореляційну матрицю.
- **Гео-візуалізація.** Choropleth-карта України з інтенсивністю тривог по областях.
- **Автоматизація.** Періодичний sync (cron/планувальник) + інкрементальне оновлення;
  кешування важких обчислень `features`/`analysis` (зараз кешується лише завантаження).
- **Збагачення зовнішніми даними.** Накладання ракетних/дронових атак (датасет
  `piterfm/massive-missile-attacks-on-ukraine`) на криву тривог.
- **Інженерія.** CI (pytest на пушах), лінтер/типізація, Docker, експорт звітів (PDF/PNG).

## Рефлексія

**Що пішло не так?**
- Не вдалося прочитати схему Kaggle-CSV автоматично (сторінка рендериться через JS),
  тож точні назви колонок були невідомі на старті.
- Під час розвідки структури API я **вичерпав ліміт** (30 запитів / 10 хв) і не міг
  одразу зробити живий sync.
- Перша вибірка реальних даних була зіпсована «вічними» тривогами окупованих територій
  (почалися у 2022 і не завершені) — часовий ряд розтягувався на 1539 днів майже з самих нулів.
- На тепловій карті `plotly` мовчки проріджував підписи осі — Дніпропетровська область
  «зникала», хоча була в даних.

**Як я скоригував підхід?**
- Замість вгадувати схему — зробив **реальні пробні запити** до API й верифікував поля та
  UID областей; для CSV додав толерантний `COLUMN_ALIASES` + крок інспекції схеми.
- Побудував **rate-limit-aware клієнт** із дисковим кешем і обробкою 429; живий sync
  запустив фоновим процесом, що дочекався відновлення вікна.
- Додав `clip_to_period()`, який відсікає «вічні» тривоги до реального вікна періоду.
- Явно задав підписи осей у `viz.heatmap` (`tickmode="array"`) і масштабовану висоту.

**Чому остаточна версія краща за першу спробу?**
- **Єдина канонічна схема подій** — обидва джерела (CSV/синтетика та live API) зводяться
  до неї, тож уся аналітика, прогноз і дашборд працюють без дублювання логіки.
- **Стійкість до реальних даних**: враховано ліміти API, вади даних, часові пояси,
  особливості рендерингу графіків — те, чого синтетична «щаслива» версія не виявляла.
- **Перевикористання та якість**: спільні `render_filters` і `viz`-білдери, дисковий кеш,
  10/10 тестів (включно з трансформацією API на реальних зразках).
